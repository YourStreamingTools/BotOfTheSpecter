## New Features & Major Changes

* **Custom Bot Mode Support**
  * Added support for running the bot with custom bot accounts via database credentials
  * New `-botusername` command line argument for specifying custom bot username
  * Automatic credential fetching from `custom_bots` table in website database
  * Comprehensive validation checks before bot initialization
  * Custom bots now show the Chat Bot badge in chat and appear under "Chat Bots" in the viewers list
  * BotOfTheSpecter maintains a separate IRC presence in the channel when running in custom mode, so it always appears in the viewers list alongside the custom bot
  * Chat sending in custom mode uses the website's shared **App Access Token** (`bot_chat_token`) with the custom bot's `bot_channel_id` as `sender_id` â€” required to keep the Chat Bot badge. The custom bot's own User Access Token is only used to maintain its IRC presence (so it appears in the viewers list).
  * **Robust IRC presence loop**: Rewrote `twitch_irc_presence()` with proper auth-failure backoff, capability-handshake handling, and notice-id classification â€” `msg_banned` / `msg_channel_blocked` / `msg_suspended` etc. trigger a 10-minute back-off, account-action notices (`msg_requires_verified_phone_number`, etc.) fail loudly with a manual-action message, and room-mode notices are silently consumed. Watch-streak IRC NOTICEs are explicitly ignored â€” the bot uses the `channel.chat.notification` EventSub event as the source of truth. Token source matrix: self mode â†’ broadcaster's `CHANNEL_AUTH`; custom mode â†’ custom bot user OAuth from `custom_bots.access_token` (auto-retries on 401); default â†’ token fetched from `twitch_bot_access` for the canonical bot id (env-var fallback). The function now accepts `override_nick` / `override_token` so a secondary BotOfTheSpecter presence can run alongside a custom bot from the same loop.

* **Per-Channel Module System**
  * New `custom_channel_modules/` package allows packaging channel-specific behavior (commands, event handlers, websocket interceptors) as Python modules without touching the core bot.
  * On startup, each module's `claims_channel(CHANNEL_NAME)` is consulted; modules that claim the active channel are instantiated and have their tables ensured via `ensure_tables()`.
  * Modules expose optional hooks: `handle_<event>(...)` coroutines (dispatched via `dispatch_module_event`), `is_module_command` / `handle_module_command`, and `is_bureau_command` / `handle_bureau_command` (dispatched via `dispatch_module_command` for every chat line).
  * `websocket_notice(..., interceptable=True)` lets a module preempt or augment internal websocket emits before they fan out.
  * Currently bundled modules: `botofthespecter` and `gfaundead`.

* **EventSub WebSocket Robustness Improvements**
  * **`session_reconnect` handling**: The bot now correctly handles server-initiated reconnects. When Twitch sends a `session_reconnect` message it includes a new WebSocket URL; the bot connects to that URL immediately (without resubscribing â€” existing subscriptions transfer automatically) instead of falling back to the hardcoded default URI. A new `EventSubReconnect` exception propagates the URL from `twitch_receive_messages()` to `twitch_eventsub()`, which sets the next connection target and loops without delay.
  * **`revocation` handling**: Subscription revocations (e.g. `chat_user_banned`, `user_removed`, `authorization_revoked`) are now detected and logged at ERROR level with the subscription type, status, and ID. When a `channel.chat.*` subscription is revoked because the bot was banned or removed from chat, an additional error log is emitted so the issue is immediately visible in logs.
  * **Session-id persistence**: Each new EventSub session id (initial connect or post-reconnect) is upserted into the `eventsub_sessions` table with `session_name = "{SYSTEM} Bot"` and `last_updated = CURRENT_TIMESTAMP`, giving the dashboard a live view of the active EventSub session.
  * **Fixed chat EventSub subscriptions failing with HTTP 400**: `channel.chat.message`, `channel.chat.notification`, and `channel.chat.user_message_hold` were being subscribed using an App Access Token, which Twitch rejects for WebSocket transport (App tokens are only valid for webhook transport). All EventSub subscriptions now always use the broadcaster's User Access Token, resolving the "invalid transport and auth combination" error.

* **Fixed Duplicate Gift Sub Chat Messages**
  * When a streamer gifted multiple subs to the community, the bot was sending one chat message and stream marker per recipient (e.g. 10 gifted subs â†’ 10 identical messages). This was caused by the `sub_gift` EventSub notification firing once per recipient in addition to the single `community_sub_gift` event.
  * Individual `sub_gift` events that belong to a community gift batch (identified by a `community_gift_id` in the payload) are now silently skipped. The `community_sub_gift` event continues to handle the single aggregate "X gifted N subs" message and stream marker as before.
  * Individually targeted gift subs (no `community_gift_id`) are unaffected and still produce their own alert.
  * The recipient-tracking dict (`gift_sub_recipients`) used to suppress duplicate sub alerts within the 30-second window is now pruned by a `cleanup_gift_sub_tracking()` background task that runs every 60 seconds, so the dict can't grow unbounded during a long stream.

## Code Improvements

* **Timer Command Refactored to Use TwitchIO Routine**
  * `!timer` no longer holds a database connection open for the entire timer duration via `await sleep()`.
  * The end-of-timer notification is now scheduled using `@routines.routine(seconds=..., iterations=1, wait_first=True)`, the same pattern used for ad-break end notices and poll half-time messages.
  * After inserting the timer record, the DB connection is released immediately; the routine fires after the requested duration, opens a fresh connection, confirms the timer is still active (guards against a concurrent `!stoptimer`), sends the end message, and cleans up.
  * `!stoptimer` now cancels the running routine (via `active_timer_routines` dict) so the end message is never sent after a manual stop.

* **Direct Per-Call Database Connections**
  * Replaced the previous shared/pooled MySQL handler with `DirectConnection`: each `mysql_connection()` call opens a fresh `aiomysql` connection and closes it on release (via async context manager), eliminating stale-connection issues across long-lived tasks.
  * `_MySQLCompat` shim preserves the old `mysql_handler.get_connection(db_name)` interface so module code keeps working unchanged.
  * Added a `cleanup_idle_db_pools()` background task placeholder that runs on a 30-minute loop (no-op; reserved for future pool reintroduction).
  * Multi-database support retained: callers pass `db_name` (user DB, `website`, `spam_pattern`).

* **Chat Message Enhancements**
  * Increased maximum message length from 255 to 500 characters
  * Updated `send_chat_message()` to support three bot modes:
    * **Self Mode** (`-self`): Uses broadcaster's own account as the bot
    * **Custom Mode** (`-custom`): Uses custom bot account credentials (with App Access Token for chat sending â€” see Custom Bot Mode Support)
    * **Standard Mode**: Uses main "botofthespecter" bot account
  * Dynamic sender_id and access_token selection based on active mode
  * Automatic BOT_USERNAME assignment (channel name for self mode, custom name for custom mode)
  * Improved error logging for failed message delivery
  * **Reliability fixes**:
    * `send_chat_message()` now uses a shared aiohttp `ClientSession` (lazy-created) to reduce transient connection churn and is closed on shutdown
    * Added retries and small backoff for transient network/server errors (502, 503, 504) and for transient aiohttp connection errors
    * Parses response body inside the request context (avoids accessing closed response objects) and uses parsed JSON/text in logs for better diagnostics
    * Force-refresh of the **website App Access Token** (not the custom bot's user token) on 401/403 responses, with a single-shot retry â€” fixes spurious send failures caused by stale shared tokens. A `_custom_token_retry_count` counter is logged per refresh.
  * **Token source via `get_website_twitch_app_credentials()`**: `send_chat_message()` (default and custom modes) now reads the App Access Token + Client ID from the website's `bot_chat_token` table, with a 60-second in-memory cache (`WEBSITE_TWITCH_CREDS_CACHE_TTL`) and force-refresh on 401/403. Env-var values remain a fallback. Keeps chat working when the website rotates the shared bot OAuth without restarting the bot.
  * **`send_long_chat_message()`**: New helper splits any response longer than 500 characters into multiple chat sends, breaking on the last whitespace before the limit (or hard-cutting at the limit if no space exists) with a 0.5-second gap between chunks. The dynamic-variable engine uses this for command output, so long commands no longer truncate at 500 characters.

* **Specter Internal WebSocket: Confirmed Registration Handshake**
  * `connect()` no longer marks the socket as connected on `REGISTER` send. The bot now waits for an explicit `SUCCESS` event from the server before flipping `websocket_connected = True`; an `ERROR` event from the server logs the rejection and disconnects.
  * Added a hard 30-second timeout on `specterSocket.connect()` to recover from servers that accept TCP but never complete the socket.io handshake (e.g. mid-reboot).
  * Reconnect backoff uses a configured delay table (`_WS_RETRY_DELAYS`) plus 0â€“3 s jitter to avoid thundering-herd reconnects after a server reboot.

* **New Internal WebSocket Event Handlers**
  * `CUSTOM_COMMAND` â€” accepts `{command, response}` from the website/API and posts the response to chat, running it through the dynamic-variable engine first so placeholders like `(game)`, `(count)`, `(json.*)` resolve. Lets the dashboard/API fire any custom command remotely.
  * `TASK_REWARD_TRIGGER` â€” when a website "task" reward fires, the bot credits the configured points via `manage_user_points`, looks up the channel's `point_name`, announces the reward in chat, and emits a `TASK_REWARD_CONFIRM` ack with the new total.
  * `SUCCESS` / `ERROR` â€” handle the new registration handshake described above.

* **TwitchIO IRC Auto-Recovery**
  * Added an `irc_health_check()` background task that polls `BOTS_TWITCH_BOT.connected_channels` every 60 seconds. If the TwitchIO IRC connection has silently dropped (no joined channels), the task logs a warning and calls `bot.connect()` to re-establish IRC so chat commands continue to work without a manual restart.

* **`safe_create_task()` for Fire-and-Forget Background Work**
  * Added a `safe_create_task(coro)` helper that wraps `asyncio.create_task()` and stores the resulting task in a module-level `_background_tasks` set with `add_done_callback` cleanup. This prevents the asyncio garbage collector from collecting in-flight tasks (which previously surfaced as "Task was destroyed but it is pending!" warnings during shutdown and under load). Most fire-and-forget call sites â€” websocket notices, alert sends, shoutout removals â€” now use it instead of bare `create_task`.

* **Event Processing**
  * New `process_chat_message_event()` helper function for consistent event handling
  * Orchestrates message counting and welcome messages; point awarding is deferred to the inner `message_counting_and_welcome_messages` `finally` block to avoid double-credit.
  * Simplified event processing from EventSub notifications
  * Provides centralized logging for all chat message events

* **Command Management Reliability**
  * Fixed `!enablecommand` and `!disablecommand` failures caused by an uninitialized `cooldown_bucket` when command configuration rows were missing.
  * Added safe cooldown defaults and kept DB lookups/updates inside an active cursor scope to prevent runtime command execution errors.
  * Improved target command parsing for these commands by normalizing input (trim, remove leading `!`, lowercase).
  * Enhanced `(customapi.*)` variable handling so placeholders inside custom API URLs (for example `(user)`) are resolved before the API request is made.
  * Improved custom API placeholder parsing to correctly handle nested parentheses in URLs, preventing truncated requests.
  * Added JSON context decoding flow for custom API placeholders:
    * `(customapi.json.<url>)` now performs a silent fetch only (never posts raw JSON or fetch errors to chat).
